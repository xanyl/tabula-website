Component Architecture & Data Flow

Server components by default, client components only when interactivity is needed

Data Flow Pattern

How Data Moves
Public Pages → No API calls
Server components render directly from Prisma queries on the server. No client-side fetching.
Mutations → Server Actions
Contact form, profile updates, inquiry creation — all via server actions with progressive enhancement. Works without JS.
Dashboard Data → Server Components + RSC
Project status, inquiry history fetched server-side. Only interactive bits (dropdowns, modals) are client components.
Auth State → Session Context
Session checked in middleware for route protection. Session data available in server components via `auth()`. Client components get it via SessionProvider.

"Client" Components Are Leaf Nodes

Principle: Push interactivity to the leaves
// Good — the form is a client leaf inside a server page
// app/(marketing)/contact/page.tsx (Server Component)
export default async function ContactPage() {
  return (
    <ContactSection>
      <ContactForm />  {/* Client Component — the only one */}
    </ContactSection>
  );
}

// Bad — making the whole page a client component
"use client";
export default function ContactPage() {
  // entire tree is now client-rendered, no SEO benefit
}
