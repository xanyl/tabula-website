Architecture Approaches

Pick the structure that best fits your scale and maintenance needs

A

Monolith Next.js App (Recommended)

Single Next.js codebase handling everything — marketing pages, auth, dashboard, API routes, and DB access — all through server components and server actions.

B

Hybrid: Next.js + Lightweight Backend

Next.js for the frontend layer, with a separate API server (Hono or tRPC) for dashboard logic. More separation but more complexity and deployment overhead.

C

Static Export + Headless CMS

Next.js static export for marketing pages, content managed via a headless CMS (Contentlayer/Notion), with a separate Express/Fastify API for auth and dashboard. Best for content-heavy sites — overkill here.

Why Option A?

Pros of Option A

  • Single deployment, single codebase — less operational overhead
  • Server components + server actions = no API boilerplate for most data operations
  • NextAuth.js integrates directly with the Next.js middleware layer
  • TypeScript end-to-end with Prisma ORM for type-safe DB access
  • Vercel deployment is seamless (or Docker on any platform)

Trade-offs

  • Next.js is a big framework — more opinions, more to learn
  • Server actions can obscure data flow if overused
  • Cold starts on serverless if not deployed to edge
