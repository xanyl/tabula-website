Key Design Decisions

Why these choices

  • ContactInquiry.userId is nullable — allows public visitors to submit without an account. Auth users get inquiry history.
  • BlogPost and CaseStudy as DB models — simpler than a CMS for this scale. Markdown content stored directly.
  • CaseStudy.metrics as JSON — flexible key-value pairs (e.g., {"time saved": "60%", "cost reduction": "$200K/yr"}) without a separate table.
  • No separate CMS table — blog posts and service descriptions are code (MDX files) or DB entries. No admin UI overhead at this stage.

Alternatives considered

  • A headless CMS (Sanity, Contentful) for blog/case studies — better editing UX but adds a paid dependency and build-time dependency. Can be added later.
  • Separate metric table for case studies — more normalized but overkill for 5-8 key-value pairs per study.
